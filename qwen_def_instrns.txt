Critical Svelte 5 Runes Rules You Must Follow Without Exception (from official docs):

Runes ($state, $derived, $effect, $props) are global functions — NEVER import them from 'svelte' or anywhere else.
- Correct: let count = $state(0);
- Wrong: import { $state } from 'svelte'; (this is invalid in stable Svelte 5)
- $derived takes a function: let doubled = $derived(() => count * 2);
- $effect for side effects: $effect(() => { console.log(count); });
- No $: reactive statements — deprecated in Svelte 5.
- Use $props() for component props when needed.
  - In Svelte 5, $props() is a global rune function that should be used only once per component to destructure all
  component props. The $props rune should only be used once in a component file.
  --Wrong:  
      let { prop1 } = $props();
      let { prop2 } =  $props();
  --Correct:
      let {prop1, prop2} = $props();
All reactivity via runes — no legacy patterns.
Event handlers: Use onclick, onkeydown (without colon) — new Svelte syntax. NOT on:click or on:keydown.
When reviewing Svelte 5 code with runes, specifically check for conditional rendering patterns using the JavaScript
  logical AND operator (&&) with JSX-style syntax. In Svelte 5, expressions like {condition && <element>} create implicit
  wrapper elements which is incorrect. Instead, Svelte uses explicit logic blocks like {#if condition}<element>{/if} for
  conditional rendering. Flag any instance where logical operators are used for conditional rendering as an error requiring
  conversion to proper Svelte logic blocks.
